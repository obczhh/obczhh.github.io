利用 JWT 漏洞：完整指南

译中：

在 JSON Web Tokens（JWTs）成为当今应用程序开发环境中主流之前，Web 应用程序主要使用服务器端会话，这带来了水平可扩展性问题。JWTs 通过将认证数据从服务器转移到令牌本身解决了这一问题。它们是自包含的、无状态的，并且经过加密签名，满足了应用程序开发中所有用例的需求。

然而，当忽视最佳实践并忽略标准安全规范以优先考虑其他问题时，JWT 漏洞就会产生，并常常导致对受影响组织和其客户构成高风险的安全缺陷。

在本文中，我们将探讨 JWT 容易受到认证绕过和注入攻击的各种方法，以此证明测试此类实现的重要性以及遵循最佳实践的有效性。

什么是 JSON Web 令牌（JWT）

JSON Web 令牌通常用于处理 Web 服务（如 Web 应用程序、API 和单页应用程序）中的身份验证会话，但它们也有其他常见用例，例如：

  • 刷新令牌以获取新的会话令牌

  • 密码重置和电子邮件确认令牌

  • 分享和邀请链接

  • 其他提供（临时）数据对象（如文档）访问权限的秘密令牌

在 API 登录后签发 JWT 的 HTTP 请求

当 JWT 的实现存在缺陷时，目标应用程序可能会受到多种 JWT 攻击，我们将在本文后续部分进行讨论。在深入利用这些令牌之前，让我们首先分解一个标准的 JSON Web Token，以便更好地帮助我们理解这些问题是如何产生的。

解析 JSON Web Tokens

按设计，JSON Web Tokens 由三部分组成：头部、载荷和签名。头部包含基本信息，用于指示库如何签名和验证令牌；载荷包含开发者决定包含在令牌中的声明和其他任意数据属性；最后，签名确保头部和载荷不能被篡改。

如上图所示，标准 JWT 令牌中的所有 3 个部分均以点（.）字符分隔，并进行 base64（URL 安全）编码。让我们更深入地了解构建有效 JSON Web Token 的每个部分。

当载荷部分被加密时，我们称之为 JSON Web 加密（JWEs）。

头部

头部包含元数据，作为 JWT 库的指令，以帮助验证、解密和签名令牌。元数据可以包括所使用的算法、令牌类型、任何标识符（例如 kid）以及其他相关元数据。

在下面的示意图中，您将找到 JWT 中可能存在的所有相关元数据属性。在本文的后续部分，我们将探讨配置错误和缺乏输入验证如何引入 JWT 漏洞。

载荷

载荷包含实际的 JWT 声明以及应用程序所需的数据。最佳实践是不包含任何敏感数据（如账单信息、电子邮件或其他个人身份信息），除非它是加密的（即 JWE）。然而，请注意，如果没有直接导致漏洞，关于缺少最佳实践的提交通常会被视为信息性发现并拒绝。

签名

签名是验证令牌完整性的必要条件。这意味着如果没有用于签名令牌的密钥，数据就无法被篡改。在本文的后续部分，我们将探讨如何忽视 JWT 实现标准会引入 JWT 漏洞。

我们现在已经了解了什么是 JSON Web Tokens（JWT），以及 JSON Web Signature（JWS）和 JSON Web Encryption（JWE）之间的区别。现在让我们深入探讨 JWT 的利用部分。

利用 JWT 漏洞

JWT 漏洞源于实施过程中的配置错误和输入验证不当。下面，我们将介绍 7 种测试 JWT 漏洞的方法。

1. 不允许使用无算法

如前文 JWT 解构部分所见，JWT 头部包含所有元数据以帮助解析 JSON Web Tokens。在某些情况下，开发者会启用无算法支持，允许他们发行和使用未签名的令牌，通常用于测试目的或启用对未签名令牌的支持（例如非敏感共享链接）。

然而，如果在生产过程中处理不当，任何人都可以篡改 JWT，包括修改任何声明以提升应用内权限或利用注入攻击。

任何用户都能够：

  1. 对头部和负载进行 Base64 解码

  2. 在头部中将 alg 属性设置为 none

  3. 篡改声明，在此情况下，我们希望将 role 属性更改为 owner 并将 organizationId 设置为受害者的组织 ID

  4. 最后，在保留尾部点的情况下完全省略签名

2. 缺失签名验证

与之前的配置错误类似，在某些情况下，当未提供签名时，开发者会完全跳过签名验证。通常，这是为了方便测试应用程序的认证部分，或为未签名令牌提供支持（例如，书签的共享链接或应用内配置预设）。

再次强调，当执行错误解析时，它将允许攻击者篡改声明，并可能冒充其他用户以提升应用内权限，或者在服务器错误处理用户输入的情况下利用注入攻击。

要测试这种情况，我们需要完全移除签名，篡改 JWT 声明，并且只将头部和负载发送到服务器。

由于这个 JWT 漏洞源于错误的 token 解析，你可能需要尝试修改’alg’头部属性，并传递意外的值，如’none’。

3. JWT 算法混淆攻击

如果所有先前尝试将算法设置为’none’都被证明是徒劳的，强烈建议进行进一步测试，因为可能提供对其他算法的支持，这可能导致 JWT 密钥混淆攻击。

JWT 算法（或密钥）混淆攻击源于开发者在处理 JWT 算法时出现错误，导致除了默认使用的算法外还支持其他算法。这允许攻击者指定任何其他算法用于令牌的签名，例如 HS256，这种算法通常使用公钥作为密钥进行签名。

在实践中，这通常涉及修改 JWT 头中的 alg 属性为 HS256，篡改负载，并使用服务器的公钥签名令牌。你需要设法找到服务器支持的精确公钥，这通常可以在服务器上找到。

4. JWK 欺骗

在其他情况下，你会发现 JWT 解析库存在缺陷，允许包含你自己的密钥对。这看似不太可能，但 CVE-2018-0114 就是一个明确的例子，它源于类型混淆，攻击者只需在令牌中包含任意的密钥对就能伪造令牌。让我们更仔细地看看这个 CVE，并考察这个 JWT 漏洞是如何被引入的。

5. 密钥 ID（kid）注入攻击

如我们所见，当头部参数处理不当时，可能会出现配置错误。在某些情况下，您可能会注意到 JSON Web Token 的头部部分引用了密钥 ID（kid）。该属性表示密钥文件的位置，本质上是指示解析库从服务器的文件系统中获取签名密钥的位置。

一些目标会使用一组密钥，因此会使用 kid 参数来识别用于签名的密钥对。在这种情况下，我们可以根据参数的处理方式来测试可能的路径遍历、SSRF（服务器端请求伪造）和注入攻击。让我们看看几个例子。

6. 强行破解常见/弱 JWT 密钥

使用常见或弱密码也可以促进伪造令牌。使用 John The Ripper 或 JWT_tool 等工具，我们可以轻易地暴力破解用于签名和验证令牌的弱密码。请注意，这只有在令牌使用密码签名时才可行。RSA 签名的 JWS 和 JWE 通常需要完整密钥对，包括私钥，这使得猜测攻击几乎不可能。

7. 列举硬编码的 JWT 密钥

有时，机密信息会意外地被推送到公共代码仓库、JavaScript 文件和其他配置文件中。这些文件可能包括用于签名和验证 JSON Web Tokens 的 JWT 密钥。

建议始终利用公开信息和侦察方法，如 JavaScript 枚举 、GitHub 搜索和 Google 挖掘 ，以发现意外硬编码的密钥。

在客户端找到 JSON Web Token 密钥是一个强有力地表明该 JSON Web Token 在客户端进行签名和验证的迹象。确保您始终验证密钥的实际机密性。
