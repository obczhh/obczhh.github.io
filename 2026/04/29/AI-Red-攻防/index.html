AI-Red-攻防

全文参考AI攻防.pdf这本书，同时也会将很多方法精简，仅供参考

用一句话总结 AI 红队和传统红队的区别,那是:我们攻击的不再是文件,而是行为。

三个根本的变化：

价值从文件引到行为：模型本身的权重是高价值的，泄露嵌入向量，输出某种训练片段，操控模型的回答引导成错误的决策。

持久化机制变动态：传统的后门，rootkit，webshell…..，现在AI的持久化为在投毒的数据里，在向量数据的某个条目中，在agent的长期记忆中，这些东西存在跨容器，会影响重新部署….

系统的自动变成主动：agent的调用工具，发邮件，触发动作等，如果一个攻击者能影响一个agent，会将其扩展为成千上万个动作。

三个必须熟悉的框架：

ATLAS、OWASP LLM Top 10、NVIDIA AI Kill Chain

MITRE ATLAS:

战术分类ATLAS是 MITRE 把 ATT&CK 思路扩展到机器学习领域的产物。它把攻击技术按战术阶段分类——训练阶段攻击、推理阶段攻击、部署阶段攻击。提供了统一标签语言。

OWASP Top 10 for LLM:

应用层风险清单，llm的专属top10：提示词注入、不安全的输出处理、训练数据污染、模型 DoS、供应链漏洞、敏感信息泄漏、不安全的插件设计、过度代理权限、过度依赖、模型窃取。

NVIDIA AI Kill Chain:

攻击者生命周期，NVIDIA 提出的 AI Kill Chain 把攻击者活动顺序化:侦察(Recon)→ 投毒(Poison)→ 劫持(Hijack)→ 持久化(Persist)→ 影响(Impact)。

这个链条的好处是它告诉防御方在哪些环节最有效地中断攻击。比如:

1、侦察阶段:加固 HTTP 元数据、移除暴露内部架构的响应头
2、投毒阶段:加摄入校验、签名验证、来源溯源
3、劫持阶段:输入消毒、模板隔离、护栏规则
4、持久化阶段:向量库完整性校验、记忆 TTL、定期清理
5、影响阶段:输出过滤、行动审批、人在回路

三框架的合作关系

1、ATLAS 提供共享词汇——你的报告里所有名词都来自 ATLAS,跨团队对齐。
2、OWASP 提供应用层测试目标——你按 OWASP 列表做检查,不会漏掉常见类别。
3、NVIDIA Kill Chain 提供时间序列——你按 Kill Chain 组织攻击和防御活动。

AI 系统是一类全新的安全目标,它的价值在于行为而非文件,持久化在于数据
而非进程,影响通过自主行动而非人手扩展。

面向 AI 目标的侦察:从端口到提示词

首先需要了解AI系统的攻击面，与传统渗透测试不一样，传统的暴露面基本本上都暴露在端口和路径中，但是AI应用不是这样的，长在七层之上——在http的自定义头字段中，在js配置文件中，在git的仓库的requirements.txt 里，在模型回答中，在RAG返回的应用源文件数据中……

AI的应用栈

概括词

用户界面层：客户端硬编码js中，endpoint，feature flag，api base path。

技术 1:直接身份探测

最朴素的方法，就是直接询问

技术 3:知识截止点测试

知识截止点(knowledge cutoff)是模型训练数据的截止日期它烧在权重里。

Ollama 里的Llama 3.2 和 Meta 服务器上的 Llama 3.2 截止点完全一样。

如果模型说”我不知道,2024 年大选还没发生”,截止点在 2024 年 11 月之前。继续往中间二分

模型不知道 GPT-4o(发布于 2024 年 5 月)→ 截止点在 2024 年初。这个时间线和 Llama 3.2 的训练时间吻合。

RAG 用相似度阈值决定何时触发检索。理解这个阈值,我们能针对性构造查询:

想最大化检索(数据外泄场景):用高相关性查询
想绕过检索(越狱场景):用低相关性查询,让 LLM 在没有”事实依据”的情况下被操纵

总结

情报不在端口里,在协议里、在元数据里、在模型行为里。

被动阶段:HTTP 头和源码仓库就能给出 80% 的架构指纹,代价为零
主动阶段:JS 配置 + 401/404 + 模型指纹 + RAG 元数据,补全运行时细节
检测规避:防御方做的是关键词匹配,不是意图分析;绕开关键词的隐蔽探测和绕开监控规则的方式高度对偶
蜜罐识别:对”太完美”的发现保持职业怀疑

攻击单个 Agent:提示词注入、记忆投毒与目标劫持

单 Agent 架构:一个会推理、能动手的 LLM

chatbot 干的事:你说一句话,它过一道 LLM,吐回一句话。一次往返,结束。

Agent 干的事:你说一句话 → 它推理”我该干什么” → 决定调用哪些工具 → 执行工具 → 看工具返回 →再推理 → 决定下一步动作 → 直到任务完成,才给你最终回答。

chatbot 最坏只能”说错话”,泄漏文本。

Agent 能读文件、查数据库、抓网页、写存储、调 API——能力范围是它被允许调用的工具集。

Agent 的五个核心组件：

循环里每一步都是潜在注入点:
1、用户消息里塞指令(直接注入)
2、工具读取的数据里塞指令(间接注入)
3、记忆里塞指令(记忆投毒)
4、工具描述里塞指令(工具描述投毒

攻防迭代周期:

  1. 枚举:摸清目标 Agent 的工具、限制、监控规则
  2. 朴素攻击:用最直接的 payload 试一下,看能不能成功
  3. 观察检测:如果失败了,看 SIEM 触发了什么规则
  4. 规避:针对性绕开规则,重新攻击

这个循环的好处是:你不光学到攻击,还学到防御方的视角。等你打完这一章,你既能写攻击 PoC,也能写检测规则。

巧法 1:重新框架(Reframing)

回 答 :”I have searched the MegacorpOne AI knowledge base using multiple relevant queries.Unfortunately, no documents matching these terms were found…”

劫持探测器没触发(没有重定向短语),但关键词密度规则触发了——查询里”internal”、”securityaudit”、”infrastructure”三个词正好踩到阈值。

巧法 2:单关键词查询

继续精简,只用一个关键词:

回答:“I found the Q4 2025 Security Audit Results in the knowledge base. Here are the criticalfindings identified during the audit conducted by PricewaterhouseCoopers LLP: ……….

完整安全审计报告吐出来了。一个关键词,无重定向短语,业务化的请求。

这一招的要点是:护栏检测的是”关键词模式”,不是”信息含量”。同样多的敏感信息,可以分散在请求里(被检测),也可以让 LLM 在响应里集中输出(检测不到)。

巧法 3:多轮渐进(Multi-turn Crescendo)

如果单关键词还不行,换多轮策略——把意图分散到多条消息里,每条单独看都无害。

第一轮(完全无害):

回答列出主题清单,我们记下 session_id 。

第二轮(同 session_id,正常 HR 查询):

返回 HR 政策。

第三轮(同 session_id,在合法对话流里偏移到安全主题):

返回完整审计报告。

关键点是 session 上下文。

每条消息单独看,关键词密度都不超阈值。

但 Agent 的对话记忆会把所有消息拼起来,形成累积上下文。

单条消息被检测,累积上下文不被检测——这是模式匹配器的根本盲
点。

系统提示词提取

系统提示词是 Agent 的”出生证”,里面通常有:身份描述、规则清单、被禁止的话题、可用工具描述、护栏关键词列表。

巧法 3:多语言切换(护栏通常只识别英文模式):

实战中三种方法可能都失败,也可能瞬间成功。

这些方法有效性取决于防护配置——这就是为什么枚举阶段要花时间摸清规则。

间接提示词注入:让数据替我说话

直接注入要求攻击者能直接和 Agent 对话。

间接注入则是把指令藏在 Agent 即将读取的数据里——这等价于 Web 安全里的”存储型 XSS”:恶意 payload 不在请求里,而在被存储的内容里,等着被无辜用户(或这里的 Agent)触发。
